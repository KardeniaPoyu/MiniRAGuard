# 一、系统概述

本系统面向在校大学生，提供合规文档的自动化法律风险审查服务。用户通过微信小程序一次性上传合同全部页面的照片，系统完成多图文字提取、法条检索与条款分析，以结构化报告的形式标注风险条款、给出处理建议，并支持基于分析结果的多轮追问对话。

系统定位：不提供正式法律咨询，不替代律师意见，作为大学生签约前的辅助参考与交互式答疑工具。

## 1.1 技术栈

| **模块**       | **技术选型**            | **说明**                                                                              |
| -------------- | ----------------------- | ------------------------------------------------------------------------------------- |
| 视觉识别       | 阿里 Qwen-VL-Max API    | 合同图片转纯文本；Prompt 内置合同鉴别逻辑，非合同图片在此阶段拦截                     |
| 法律推理       | DeepSeek V3 API         | 低成本强推理，强制 JSON 输出，单次调用成本约 0.03 元                                  |
| 向量检索       | LlamaIndex + ChromaDB   | RAG 标准搭配，本地持久化，无外部服务依赖                                              |
| Embedding 模型 | BAAI/bge-small-zh-v1.5  | 中文向量模型，完全免费，首次运行时自动下载（约 100 MB）                               |
| 缓存           | SQLite（Python 标准库） | MD5 文本哈希查重，命中时零成本、毫秒级响应，无新增依赖                                |
| 后端           | Python + FastAPI        | 异步框架，自动生成接口文档，负责 serve 前端静态文件                                   |
| 前端           | UniApp + HBuilderX      | 微信小程序形态，一套代码编译至小程序，HBuilderX 为官方 IDE；支持多图选择与多轮对话 UI |

## 1.2 前端技术说明

UniApp 为前端框架（基于 Vue.js），HBuilderX 为对应 IDE，二者配合使用，最终编译产物为微信小程序包。两者不存在选择关系，均为必要工具。

前端开发过程中需注意以下微信小程序环境限制：

- 网络请求须使用 uni.request()，不可使用浏览器原生 fetch 或 XMLHttpRequest；
- 多图选取通过 uni.chooseImage() 的 count 参数控制上限（建议设为 10），每张图片经 uni.getFileSystemManager().readFile() 转换为 base64 字符串后，以数组形式传输；
- 多轮对话的历史记录维护在前端 data 中，每次追问时将完整历史随请求一并发送；
- 本地调试阶段在微信开发者工具中开启「不校验合法域名」，上线前须关闭并配置正式域名。

# 二、数据流

## 2.1 合同审查流程（POST /api/analyze）

用户一次性上传合同全部页面后，后端按以下顺序执行处理：

POST /api/analyze ← { images: \[base64_p1, base64_p2, ...\] }

Step 1 对 images 数组中每张图片循环调用 vision_tool.extract_contract()

→ 任意一张 is_contract: false → 返回错误响应，终止流程

→ 全部 is_contract: true → 各页文本按顺序拼接为 contract_text

Step 2 cache_tool.get_cache(contract_text)

计算 MD5，查询 SQLite

→ 命中 → 直接返回历史 JSON，终止流程

→ 未命中 → 继续

Step 3 rag_tool.retrieve_legal_context(contract_text)

双路检索：语义向量 Top-3 + 关键词强制召回

→ 输出法条上下文字符串

Step 4 reviewer.review_contract(contract_text, legal_context)

组装 Prompt → 调用 DeepSeek V3 API

→ 解析并验证 JSON 结构

Step 5 cache_tool.set_cache(contract_text, result)

写入 SQLite 缓存

→ 返回 JSON 响应至前端

## 2.2 多轮对话流程（POST /api/chat）

审查结果返回后，用户可在对话框中追问，后端将原始分析结果作为上下文注入 Prompt：

POST /api/chat ← { question: str, context: dict, history: list }

context = 第一次 /api/analyze 返回的完整 JSON

history = \[ {role, content}, ... \] 前端维护，每轮追加

Step 1 chat_tool.build_chat_prompt(question, context, history)

将 context 序列化后注入系统 Prompt

拼接 history 构建多轮消息列表

Step 2 调用 DeepSeek V3 API（非结构化自由回复）

→ 返回 { answer: str } 至前端

→ 前端将本轮问答追加至 history

## 2.3 审查接口响应结构

DeepSeek 返回结果经 json.loads() 验证后，原样透传至前端。original_text 字段同时用于前端文本高亮（匹配后以黄色背景标注原句，替代图像标注方案）：

{

"overall_risk": "高风险",

"summary": "发现 2 处风险条款，建议签署前与房东协商修改。",

"analysis_results": \[

{

"clause_id": 1,

"original_text": "租客若提前退租，押金概不退还",

"risk_level": "高风险",

"risk_type": "霸王条款",

"reason": "押金具有担保性质，房东仅可扣除实际损失，不得全额没收。",

"legal_basis": "《民法典》第七百零四条",

"advice": "建议改为：提前 30 天书面通知，押金扣除实际损耗后退还。"

}

\]

}

字段约束：risk_level 取值为「高风险」「中风险」「低风险」三种；risk_type 为自由文本；legal_basis 格式固定为「《法规名称》第X条」。

文本高亮说明：前端在渲染合同全文时，对 original_text 字段的内容进行字符串匹配，命中文本以高亮样式呈现，用户可直接在合同文本中定位风险条款，无需人工逐行对照卡片。
